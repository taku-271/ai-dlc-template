# コードレビュー指摘の理解・修正・学習フロー

引数としてPR番号を受け取ります（例: `/review-apply 42`）。

「gh でレビューを取得 → 開発者が指摘を理解 → TDD で修正 → .learnings/ に保存」の順で進みます。

---

## モデル戦略

| ステップ | モデル | 理由 |
|---|---|---|
| Step 1 レビュー取得・整理 | メインセッション | 指摘の分析・分類が必要 |
| Step 2 理解確認 | メインセッション | 人間との対話が必要 |
| Step 3 fix.md 生成 | `sonnet`（Agent ツール） | 構造化ドキュメント生成 |
| Step 4 テスト作成 + RED確認 | `sonnet`（Agent ツール） | テスト生成からRED確認まで完結 |
| Step 5 修正実装 | メインセッション | 複雑な実装判断が必要 |
| Step 6 .learnings/ 保存 | `sonnet`（Agent ツール） | ファイル生成のみ |

---

## Step 1: レビューコメントの取得・整理

以下のコマンドでPRのレビュー内容を取得してください：

```bash
# リポジトリの owner/name を取得
REPO=$(gh repo view --json owner,name -q '"\(.owner.login)/\(.name)"')

# PRの概要
gh pr view [PR番号]

# インラインコメント（ファイル単位の指摘）
gh api "repos/$REPO/pulls/[PR番号]/comments" \
  --jq '.[] | {path: .path, line: .original_line, body: .body, user: .user.login}'

# レビュー本文（REQUEST_CHANGES・APPROVED のコメント）
gh api "repos/$REPO/pulls/[PR番号]/reviews" \
  --jq '.[] | select(.state == "REQUEST_CHANGES" or (.body | length > 0)) | {state: .state, body: .body, user: .user.login}'
```

取得した指摘を以下の観点で分類してください：

| 分類 | 説明 | .learnings/ 保存先 |
|---|---|---|
| **pattern** | 命名規則・コーディング規約・設計パターン | `.learnings/patterns/` |
| **error** | バグ・ロジックミス・セキュリティ問題 | `.learnings/errors/` |
| **preference** | レビュアーの好み・スタイル | `.learnings/preferences/` |
| **context** | アーキテクチャ・ドメイン知識 | `.learnings/context/` |

---

## Step 2: 開発者への提示と理解確認（必須チェックポイント）

以下の形式で指摘一覧を提示してください：

```
📋 PR #[番号] のレビュー指摘：[件数]件

─────────────────────────────────────────
[1] [pattern / error / preference / context]
場所: ファイル名:行番号（なければ「全体」）
指摘: レビュアーのコメント原文
なぜ問題か: AIによる解説 — 放置するとどうなるか・なぜ良くないか
─────────────────────────────────────────
[2] ...
```

提示後、以下のメッセージを出力して必ず停止してください：

「各指摘について、ご自身の言葉で「なぜ問題か」を教えてください。
理解を確認してから修正に進みます。

⚠️ 「全部わかった、進めて」はNGです。指摘ごとに一言ずつお願いします。」

**開発者が各指摘について自分の言葉で説明するまで次に進まないこと。**

---

## Step 3: fix.md の生成（理解確認後に実行）

**Agent ツール（model: `sonnet`）を使ってサブエージェントで実行してください。**

`docs/reviews/YYYY-MM-DD-pr[PR番号]-fix.md` を以下の構成で生成する（日付は本日）：

```markdown
# PR #[番号] レビュー指摘対応

## 対象PR
- URL: [PR URL]
- レビュアー: [名前]
- 日付: [YYYY-MM-DD]

## 指摘一覧と対応方針

### [1] [分類] ファイル名:行番号
- **指摘**: [内容]
- **根本原因**: [なぜ起きているか]
- **対応方針**: [どう直すか]

### [2] ...

## 変更予定ファイル
- `src/...`
```

生成後、以下のメッセージを出力して必ず停止してください：

「✅ fix.md を生成しました。`docs/reviews/[ファイル名]` をレビューしてください。
問題なければ「OK」とお伝えください。」

**「OK」があるまで次のステップに進まないこと。**

---

## Step 4: テストの作成と RED 確認（OK 後に実行）

**Agent ツール（model: `sonnet`）を使ってサブエージェントで実行してください。**

- fix.md の各指摘に対応するテストを作成する
- テストは「指摘が修正された正しい状態」を表現すること（バグ再現ではなく期待動作を定義）
- テストを実行して全件 RED であることを確認する
- 報告: 作成したテストファイル・RED件数・RED でないテストがあれば原因

---

## Step 5: 修正実装（メインセッションで実行）

テストが GREEN になるよう修正してください。

- `docs/inception/security-policy.md` のルールを守ること
- 修正は根本原因に対して行う（症状を隠す workaround にしない）
- 既存テストが壊れていないか確認すること
- 全テストが GREEN になったら Agent ツール（model: `haiku`）で確認

---

## Step 6: .learnings/ への自動保存（Agent ツール `sonnet`）

各指摘の分類に応じて `.learnings/` に保存してください。

ファイル命名: `.learnings/[分類]/[kebab-case-topic].md`

保存形式（既存ファイルがあれば追記、なければ新規作成）：

```markdown
# [指摘の要約タイトル]

- **問題**: 何が起きていたか・なぜ問題か
- **学び**: 正しいアプローチ
- **出典**: PR #[番号] by [レビュアー] ([YYYY-MM-DD])
```

**既存の `.learnings/` ファイルと重複する場合は新規作成せず追記すること。**

---

## Step 7: 完了報告（Final Report）

| 項目 | 内容 |
|---|---|
| 対応した指摘 | 件数・内容の要約 |
| 変更ファイル | 影響範囲（追加・変更） |
| テスト結果 | 追加テスト件数・全体 GREEN/RED |
| .learnings/ 更新 | 保存・追記したファイル一覧 |
| 未対応の指摘 | スキップした指摘とその理由 |
