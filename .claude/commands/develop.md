# 機能開発フロー（全ステップ通し）

引数として機能名を受け取ります（例: `/develop user-auth`）。

spec.md の生成 → レビュー待ち → test-spec.md の生成 → レビュー待ち → 実装 の順で進みます。

---

## Step 1: spec.md の生成

`docs/_templates/feature.md` をベースに `docs/features/[機能名]/spec.md` を生成してください。

生成前に以下を読むこと：
- `docs/inception/requirements.md`
- `docs/inception/domain-model.md`
- `docs/inception/security-policy.md`
- `CLAUDE.md`（技術スタックの確認）
- `docs/features/` 配下の既存機能（命名規則・粒度の参考）
- `.learnings/` 配下（過去の学びを適用）

不明な点は生成前にまとめて質問してください。

生成完了後、以下のメッセージを出力して必ず停止してください：

「✅ spec.md を生成しました。`docs/features/[機能名]/spec.md` をレビューしてください。修正があれば直接ファイルを編集してください。問題なければ「OK」とお伝えください。」

**「OK」という返答があるまで次のステップに進まないこと。**

---

## Step 2: Material Ambiguity の確認（OK をもらってから実行）

実装の方向・権限・ユーザー可視動作に影響する **曖昧な点** を洗い出して確認してください。

以下の観点で spec.md を読み直し、「推測で進めると後工程で手戻りが発生する」箇所を特定する：
- 実装方針が複数考えられる箇所
- 権限・認可の扱いが不明な箇所
- ユーザーに見える動作（エラーメッセージ・リダイレクト先等）が未定義な箇所
- データの扱い（更新か上書きか・削除時の挙動等）が曖昧な箇所

曖昧点がある場合はまとめて質問し、回答を spec.md の「未解決事項」に反映してから次に進む。
曖昧点がない場合はその旨を報告して次のステップに進む。

---

## Step 3: test-spec.md の生成（OK をもらってから実行）

`docs/_templates/test-spec.md` をベースに `docs/features/[機能名]/test-spec.md` を生成してください。

生成前に最新の `docs/features/[機能名]/spec.md` を読むこと。

**テストケースは「成功の定義」を言語化する作業**。何をもって実装完了とするかを先に決める。

含めるテストケース：
- ハッピーパス（正常系）
- 異常系・エラーケース
- 権限・認証に関するセキュリティケース（`docs/inception/security-policy.md` を参照）
- spec.md に記載されたパフォーマンス要件の検証

生成完了後、以下のメッセージを出力して必ず停止してください：

「✅ test-spec.md を生成しました。`docs/features/[機能名]/test-spec.md` をレビューしてください。修正があれば直接ファイルを編集してください。問題なければ「OK」とお伝えください。」

**「OK」という返答があるまで次のステップに進まないこと。**

---

## Step 4: 実装（OK をもらってから実行）

### 4-1: 実装計画の提示

コードを書く前に以下を提示して承認を得てください：
- 作成・変更するファイルの一覧（`docs/_templates/implementation-draft.md` 形式）
- 採用するAPI層（tRPC / GraphQL）
- 特記事項・既知の落とし穴

**承認後に実装に進むこと。**

### 4-2: テストの実装（先に必ずテストを書くこと）

**⚠️ テストファイルを作成するまで、実装コードを1行も書かないこと。**

`test-spec.md` の全テストケースをテストコードに落としてください。
この時点では実装コードが存在しないため、テストは必ず失敗（RED）します。

テストファイルを作成したら、テストを実行して全件 **RED（失敗）** であることを確認してください。

完了後、以下のメッセージを出力して必ず停止してください：

「✅ テストファイルを作成しました。テストを実行して全件 RED になっていることを確認してください。確認できたら「テストOK」とお伝えください。」

**「テストOK」という返答があるまで実装コードを1行も書かないこと。**

### 4-3: 実装（「テストOK」をもらってから実行）

テストが GREEN になるように実装してください。

1. **型定義・Schema**
2. **バックエンド実装**（`security-policy.md` のルールを守ること）
3. **フロントエンド実装**
4. **全テストが GREEN になっていることを確認する**

### 4-4: status.md の更新

`docs/features/[機能名]/status.md` を作成・更新してください：

```markdown
# [機能名] 実装状況

## ステータス: 実装完了

## 実装済みファイル
- `src/...`

## テスト結果
- [ ] E2Eテスト: X件 passed
- [ ] 型チェック: passed

## spec.md との差異
なし / あり（内容を記載）
```

### 4-5: 完了報告（Final Report）

以下の形式で報告してください：

| 項目 | 内容 |
|---|---|
| 変更概要 | 何を実装したか |
| 変更ファイル | 影響範囲（追加・変更・削除） |
| テスト結果 | 件数・結果（GREEN/RED） |
| spec.md との差異 | なし / あり（内容を明記） |
| リスク・懸念点 | 副作用・今後の注意点 |
| 未完了事項 | 残タスク（あれば） |
