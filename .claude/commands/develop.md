# 機能開発フロー（全ステップ通し）

引数として機能名を受け取ります（例: `/develop user-auth`）。

spec.md の生成 → レビュー待ち → test-spec.md の生成 → レビュー待ち → 実装 の順で進みます。

## モデル戦略（トークン削減）

各ステップで使用するモデルを以下の通り使い分けてください：

| ステップ | モデル | 理由 |
|---|---|---|
| spec.md・test-spec.md の生成 | `sonnet`（Agent ツール） | 構造化ドキュメント生成。深い推論不要 |
| Material Ambiguity の反映・実装計画 | メインセッション（Opus） | トレードオフ評価・設計判断が必要 |
| テストコードの実装 + RED確認（1回） | `sonnet`（Agent ツール） | テスト生成からRED確認まで1エージェントで完結 |
| コア実装（バックエンド・フロント） | メインセッション（Opus） | 複雑な実装判断が必要 |
| GREEN 確認 | `haiku`（Agent ツール） | Bash 実行と結果読み取りのみ |

---

## Step 1: spec.md の生成

**Agent ツール（model: `sonnet`）を使ってサブエージェントで実行してください。**

サブエージェントへの指示: `docs/_templates/feature.md` をベースに `docs/features/[機能名]/spec.md` を生成する。

生成前に以下を読むこと：
- `docs/inception/requirements.md`
- `docs/inception/domain-model.md`
- `docs/inception/security-policy.md`
- `CLAUDE.md`（技術スタックの確認）
- `docs/features/` 配下の既存機能（命名規則・粒度の参考）
- `.learnings/` 配下（過去の学びを適用）

加えて、実装の方向・権限・ユーザー可視動作に影響する曖昧な点（Material Ambiguity候補）を洗い出して要確認事項にまとめること。

**報告内容は以下のみ（詳細な作業ステップは不要）：生成したファイルのパス・要確認事項・Material Ambiguity候補**

生成完了後、以下のメッセージを出力して必ず停止してください：

「✅ spec.md を生成しました。`docs/features/[機能名]/spec.md` をレビューしてください。修正があれば直接ファイルを編集してください。

また、以下の曖昧点についても合わせてご確認ください：
[サブエージェントの要確認事項・Material Ambiguity候補を列挙]

spec.md の確認と曖昧点への回答をまとめて「OK」とお伝えください。」

**「OK」という返答があるまで次のステップに進まないこと。**

---

## Step 2: Material Ambiguity の反映（OK をもらってから実行）

Step 1 で人間から受け取った曖昧点への回答を `spec.md` の「未解決事項」に反映する。

Step 1 で挙げられなかった追加の曖昧点があれば質問してから次に進む。
追加の曖昧点がなければそのまま次のステップに進む。

---

## Step 3: test-spec.md の生成（OK をもらってから実行）

**Agent ツール（model: `sonnet`）を使ってサブエージェントで実行してください。**

サブエージェントへの指示: `docs/_templates/test-spec.md` をベースに `docs/features/[機能名]/test-spec.md` を生成する。

生成前に最新の `docs/features/[機能名]/spec.md` と `docs/inception/security-policy.md` を読むこと。

**テストケースは「成功の定義」を言語化する作業**。何をもって実装完了とするかを先に決める。

含めるテストケース：
- ハッピーパス（正常系）
- 異常系・エラーケース
- 権限・認証に関するセキュリティケース（`docs/inception/security-policy.md` を参照）
- spec.md に記載されたパフォーマンス要件の検証

**報告内容は以下のみ（詳細は不要）：生成したファイルのパス**

生成完了後、以下のメッセージを出力して必ず停止してください：

「✅ test-spec.md を生成しました。`docs/features/[機能名]/test-spec.md` をレビューしてください。修正があれば直接ファイルを編集してください。問題なければ「OK」とお伝えください。」

**「OK」という返答があるまで次のステップに進まないこと。**

---

## Step 4: 実装（OK をもらってから実行）

### 4-1: 実装計画の提示

コードを書く前に以下を提示して承認を得てください：
- 作成・変更するファイルの一覧（`docs/_templates/implementation-draft.md` 形式）
- 採用するAPI層（tRPC / GraphQL）
- 特記事項・既知の落とし穴

**承認後に実装に進むこと。**

### 4-2: テストの実装（先に必ずテストを書くこと）

**⚠️ テストファイルを作成するまで、実装コードを1行も書かないこと。**

**Agent ツール（model: `sonnet`）を使ってサブエージェントで実行してください。**

サブエージェントへの指示: `test-spec.md` の全テストケースをテストコードに落とし、そのままテストを実行して全件 RED（失敗）であることを確認する。報告内容は以下のみ（詳細は不要）：作成したテストファイル一覧・RED件数・RED でないテストがあれば原因。

- 全件 RED であれば次の 4-3 に進む
- RED でないテストがある場合は原因を調査・修正してから進む

### 4-3: 実装

**このステップはメインセッション（Opus）で実行してください。**

テストが GREEN になるように実装してください。

1. **型定義・Schema**
2. **バックエンド実装**（`security-policy.md` のルールを守ること）
3. **フロントエンド実装**
4. **全テストが GREEN になっていることを確認する**（Agent ツール（model: `haiku`）でテストを実行して確認。報告は件数と結果のみ）

### 4-4: status.md の更新

`docs/features/[機能名]/status.md` を作成・更新してください：

```markdown
# [機能名] 実装状況

## ステータス: 実装完了

## 実装済みファイル
- `src/...`

## テスト結果
- [ ] E2Eテスト: X件 passed
- [ ] 型チェック: passed

## spec.md との差異
なし / あり（内容を記載）
```

### 4-5: 完了報告（Final Report）

以下の形式で報告してください：

| 項目 | 内容 |
|---|---|
| 変更概要 | 何を実装したか |
| 変更ファイル | 影響範囲（追加・変更・削除） |
| テスト結果 | 件数・結果（GREEN/RED） |
| spec.md との差異 | なし / あり（内容を明記） |
| リスク・懸念点 | 副作用・今後の注意点 |
| 未完了事項 | 残タスク（あれば） |
