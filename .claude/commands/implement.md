# 設計書からコードを実装する

引数として機能名を受け取ります（例: `/implement user-auth`）。

## Step 1: 設計書の読み込み

以下を読んでください：
- `docs/features/[機能名]/spec.md`
- `docs/features/[機能名]/test-spec.md`
- `docs/inception/security-policy.md`
- `CLAUDE.md`
- `.learnings/` 配下（過去の学びを適用）

spec.md または test-spec.md に「未解決事項」がある場合は実装前に確認してください。

## Step 2: Material Ambiguity の確認

実装前に以下の観点で曖昧な点を洗い出してください：
- 実装方針が複数考えられる箇所
- 権限・認可の扱いが不明な箇所
- ユーザーに見える動作（エラーメッセージ・リダイレクト先等）が未定義な箇所
- データの扱い（更新か上書きか・削除時の挙動等）が曖昧な箇所

曖昧点がある場合はまとめて質問してから次に進む。

## Step 3: 実装計画の提示

コードを書く前に以下を提示して承認を得てください：
- 作成・変更するファイルの一覧（`docs/_templates/implementation-draft.md` 形式）
- 採用するAPI層（tRPC / GraphQL）
- 特記事項・既知の落とし穴

**承認を得てから実装に進むこと。**

## Step 3: テストの実装（先に必ずテストを書くこと）

**⚠️ テストファイルを作成するまで、実装コードを1行も書かないこと。**

`test-spec.md` の全テストケースをテストコードに落としてください。
この時点では実装コードが存在しないため、テストは必ず失敗（RED）します。

テストファイルを作成したら、テストを実行して全件 **RED（失敗）** であることを自分で確認してください。

- 全件 RED であれば次の Step 4 に進む
- RED でないテストがある場合は原因を調査・修正してから進む

## Step 4: 実装

テストが GREEN になるように実装してください。

### 4-1: 型定義・Schema

- tRPC の場合: `src/server/routers/[機能名].ts`
- GraphQL の場合: `src/graphql/[機能名].graphql`

### 4-2: バックエンド実装

ビジネスロジック・データアクセス層を実装してください。
`security-policy.md` のルールを必ず守ること。

### 4-3: フロントエンド実装

コンポーネント・APIクライアントを実装してください。
CLAUDE.md のフレームワーク（Next.js / Vite）に合わせること。

### 4-4: テストが GREEN になっていることを確認する

## Step 5: status.md の更新

`docs/features/[機能名]/status.md` を更新してください：

```markdown
# [機能名] 実装状況

## ステータス: 実装完了

## 実装済みファイル
- `src/...`

## テスト結果
- [ ] E2Eテスト: X件 passed
- [ ] 型チェック: passed

## spec.md との差異
なし / あり（内容を記載）
```

## Step 6: 完了報告（Final Report）

以下の形式で報告してください：

| 項目 | 内容 |
|---|---|
| 変更概要 | 何を実装したか |
| 変更ファイル | 影響範囲（追加・変更・削除） |
| テスト結果 | 件数・結果（GREEN/RED） |
| spec.md との差異 | なし / あり（内容を明記） |
| リスク・懸念点 | 副作用・今後の注意点 |
| 未完了事項 | 残タスク（あれば） |

## 重要なルール
- spec.md・test-spec.md に書いていないことは実装しない
- E2Eテストが先、実装が後（TDD）
- セキュリティルールは必ず守る
