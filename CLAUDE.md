## 学習システム

作業を始める前に `.learnings/` 配下を確認し、過去の学びを適用する。

人間に指摘・修正されたとき、新しいパターンを発見したとき、即座に `.learnings/` を更新する：
- ミス・エラー → `.learnings/errors/[種類].md`
- コードパターン・慣習 → `.learnings/patterns/[名前].md`
- 人間の好み・判断基準 → `.learnings/preferences/[テーマ].md`
- プロジェクト固有の文脈・背景 → `.learnings/context/[テーマ].md`

## 鉄則（必ず守ること）

- コードを書く前に必ず `docs/features/[機能名]/spec.md` を読む
- 設計書にない仕様を勝手に決めない。不明な点は必ず質問する（Material Ambiguity発見時は規模によらず必ず質問して停止する）
- 実装計画は人間に提示して承認を得てから進める。ただし `/feature-flow` で **Small** と判定したタスクは承認待ちを省略し、完了報告での事後レビューでよい（`/feature-flow` の Small自動モードを参照）。**Medium/Large** は引き続き事前承認を得る
- 実装規模によらず `docs/features/[機能名]/spec.md`・`test-spec.md`・`status.md` は必ず作成する（ドキュメント作成の省略は不可）
- 設計書と実装に差異が出た場合は必ず報告する
- 実装中に不明点・不整合を発見した場合は即座に作業を止めて報告する
- `docs/inception/security-policy.md` のルールを常に守る
- `docs/inception/domain-model.md` の用語定義を必ず参照する
- `docs/inception/requirements.md` のスコープ外の実装はしない

## NEVER（絶対にやってはいけないこと）

以下の操作は **人間の明示的な指示がない限り実行禁止**：

| カテゴリ | 禁止操作 | 理由 |
|---|---|---|
| 破壊的ファイル操作 | `rm -rf`・`git reset --hard`・`git clean -f` | 復元不可能なデータ消失 |
| DB破壊 | `DROP TABLE`・`DELETE` without `WHERE`・`TRUNCATE` | 大規模データ消失 |
| 本番操作 | 本番環境へのデプロイ・本番DBへの直接接続 | AIはインシデント状況を認識できない |
| 認証・セキュリティ | 権限チェックロジックの削除・トークン検証のスキップ | バグが数ヶ月後に発覚する |
| Secrets操作 | `.env` の読み取り・APIキーのログ出力 | 情報漏洩リスク |
| インフラ変更 | `terraform destroy`・IAM権限変更・kubectl delete | 環境破壊・セキュリティホール |

**元に戻せない操作には必ず人間のチェックポイントを設ける。**
